# GaussDB ETL Builder Skill — 设计文档

日期:2026-06-16
状态:已确认,待编写

## 背景与目标

`ddh-agent` 的 Java 后端(`AgentDomainService`)已经实现了"平台分析需求 → 完整 ETL 作业"的能力:通过状态机 + LLM 工具调用 + 人工确认,把需求变成目标表 schema、ETL 步骤计划,并逐步生成可执行的 GaussDB SQL,产出 `plan.md` 和每步一个 `.sql` 文件。

本设计的目标是把这一能力沉淀成一个**跨模型可移植、可安装的 SKILL.md 技能包**,供 Qwen 等其它大模型使用。技能以**交互式引导**的方式工作:分析需求、逐步思考、引导用户确认,最终生成完整 ETL 作业并写入文件。

### 非目标
- 不替换或修改现有 Java 后端实现。
- 不做多方言支持(锁定 GaussDB)。
- 不依赖 ddh-agent 的数据库或后端工具;源表信息由用户在对话中提供。

## 关键决策(已与用户确认)

| 维度 | 决策 |
|---|---|
| 形态 | Anthropic SKILL.md 格式(纯 Markdown 指令包),可被 Qwen 等模型加载 |
| 工作模式 | 交互式引导,逐步确认 |
| 源表来源 | 模型**主动向用户索要**建表语句/字段清单(缺什么问什么) |
| SQL 方言 | **锁定 GaussDB**(PostgreSQL 兼容) |
| 产物 | **必须写文件**:`plan.md` + 每步 `stepN_<名称>.sql` |
| 结构路线 | 路线 B(带兜底):主流程 + 引用文件,硬约束保留在主流程 |

## 结构(路线 B)

```
gaussdb-etl-builder/
├── SKILL.md                              # 工作流 + GaussDB 硬约束
├── references/
│   ├── gaussdb-sql.md                    # 方言细节、DDL 模板、转换套路、完整样例
│   └── platform-variables-example.md     # 平台变量文档的形态示范(${TX_DATE} 等)
└── assets/
    ├── plan-template.md                  # plan.md 模板
    └── knowledge-template.md             # 沉淀知识文档的模板
```

可移植性兜底:GaussDB 的**关键硬约束**(文件命名规则、目标表/临时表创建顺序、CREATE TABLE/TEMP TABLE 必带等)留在 `SKILL.md`,保证即使宿主不自动加载引用文件也不丢核心约束;篇幅大的语法示例放 `references/`,并在 SKILL.md 中显式指示"生成 SQL 前先读取该文件"。

## 元信息

- **name**: `gaussdb-etl-builder`
- **description**(触发导向,写得主动一些):当用户想把一个数据分析需求变成可执行的数仓 ETL 作业、生成 GaussDB 建表/清洗/装载 SQL、规划 ETL 步骤时使用——即使用户没明说 "ETL" 也应触发(例如"统计各科前十名生成一张报表表""把这几张源表加工成一张宽表")。

## 工作流(四阶段,对齐 Java 版状态机)

| 阶段 | 模型做什么 | 闸门 |
|---|---|---|
| **P0 开场** | 复述对需求的理解;若用户未提供源表,**主动索要**相关源表的建表语句/字段清单(名称、类型、注释);询问是否有**外挂知识文档**与**平台变量文档**并按需加载;确认**输出目录**(默认 `./etl/<需求短名>/`) | — |
| **P1 定目标表** | 分析需求 → 提出目标表名 + 字段(名/类型/注释)→ 清晰展示 | **停下等用户确认**,未确认不向下推进 |
| **P2 定步骤计划** | 规划完整步骤:① 建目标表 ② 各清洗/转换步 ③ 需要时建临时表;保证"先建后写"顺序;若加载了平台变量文档且需求含日期/增量语义,在此考虑全量 vs 增量装载 | **停下等用户确认** |
| **P3 生成 SQL + 落盘** | 按确认计划逐步生成完整可执行 GaussDB SQL;**写文件** `plan.md` + 每步 `stepN_<名称>.sql`;套用平台变量(若有);汇报写了哪些文件;最后汇总本次确认的知识、征得同意后写回知识文档 | — |

任意阶段用户修改需求 → 回到对应阶段重新推导。

## 文件产物约定(沿用现有格式)

- **目录**:P0 确认的输出目录;不存在则创建。
- **`plan.md`** 结构(沿用 Java `writePlanMd`):
  - `# ETL 执行计划`
  - `## 需求描述`
  - `## 目标表`(反引号包裹表名)
  - `## ETL 步骤`,每步:`### Step N: <名称>（临时表）`(临时表才带后缀)、描述、`输出表：<表名>`
- **SQL 文件**:`step<序号>_<安全名>.sql`,UTF-8 编码。安全名规则同 Java:非 `[a-zA-Z0-9_中文]` 字符替换为 `_`,超过 64 字符截断。

## 数据区命名规则(2026-06-16 真实使用反馈补充)

平台要求落地表建在指定数据区。用户在 P0 提供一个 **4 位数据区编码**(数据库编码,如 `ABCD`),由它推导:

- **Schema** = `<编码>_DATA`(如 `ABCD_DATA`)
- **表名前缀** = `<编码>_`(如 `ABCD_`),该前缀**取代**普通语义前缀(不再写 `rpt_`/`dws_`)

目标表/持久中间表用 schema 限定全名 `<编码>_DATA.<编码>_<业务名>` 创建并引用(如 `ABCD_DATA.ABCD_EXAM_TOP10_BY_SUBJECT`);源表按用户给定名引用。**临时表例外**:`CREATE TEMPORARY TABLE` 走会话临时 schema,不加 schema、不加前缀。落地点:P0 索要编码、P1 用全名提出目标表、P3 全名建表与引用、references 模板与 worked example、plan 模板新增「数据区」节。

**标识符强制大写**:生成 SQL 里数据区编码、schema、表名、字段名一律大写(源表引用也大写,靠未加引号折叠);CTE 名与表别名属局部名保持小写;中文注释、步骤文件名不受影响。

**字段类型白名单(可配置)**:平台支持的字段类型有限,新增 `references/supported-column-types.md` 作为唯一权威类型清单,使用者可按自己平台增删。P1 选型、P3 建表只能用表内类型;`gaussdb-sql.md` 不再另立类型表,改为指向白名单。

## GaussDB 硬约束(置于 SKILL.md 主流程,保证不丢)

- PG 兼容语法;SQL 必须完整、可直接执行。
- 建目标表的步骤:`is_temp_table=false`,必须 `CREATE TABLE`,使用确认后的 schema。
- 临时表:`CREATE TEMPORARY TABLE`,**仅在确实需要时**才创建;不需要则完全不出现。
- **顺序铁律**:所有临时表与目标表必须在任何写入它们的步骤**之前**创建。
- 清洗/装载步使用 `INSERT INTO ... SELECT` 等真正写入目标/临时表的语句。

## references/gaussdb-sql.md(篇幅大的细节)

- GaussDB/PG 类型映射与常见坑(`VARCHAR` 长度、`NUMERIC`、日期函数等)
- `CREATE TABLE` / `CREATE TEMPORARY TABLE` 模板(含可选 `DISTRIBUTE BY` 说明)
- 常见转换套路:窗口函数排名(`ROW_NUMBER() OVER`)、去重、多表 JOIN
- **一个完整 worked example**:复用 `backend-java/projects/3` 的"各科前十名"案例(需求 → 目标表 → 分步 SQL),作为模型样板
- SKILL.md 在 P3 开头显式指示:生成 SQL 前先读取本文件

## 外挂知识文档:表语义 & 字段关联(可读可写)

沉淀**已确认**的领域知识以跨会话复用:
- 每张表的用途
- 表间字段关联(`A.col ↔ B.col`)及基数(`1:1` / `1:N`)
- (可选)字段口径、枚举值含义

**加载**:P0 主动询问"是否有已沉淀的知识文档?有就给路径";提供则读取,并在 P1/P2 用它减少重复提问(已知 join key、基数则不再问用户)。

**写回(沉淀)**:本次会话用户新确认的表用途/关联/基数,在 **P3 完成后**汇总,**先展示变更摘要、经用户确认后**再写入/更新该文档。无现成文档时,主动提议代建一份。

**范围**:一份按"数据域/项目"**持续累积**的文档(开场读取、末尾增量更新),而非每次会话新建。

**格式**:固定模板 `assets/knowledge-template.md`,保证写出的文档下次仍可被同一套规则读回。结构示意:

```
## 表:<表名>
用途:...
字段:<名> <类型> — <口径>

## 关联关系
<A.col> ↔ <B.col>  [1:N]  说明:...
```

## 外挂平台变量文档:把 ${TX_DATE} 这类语法融入 SQL

平台执行期支持的变量(如 `${TX_DATE}` = 作业实例执行日的前一天)通过外挂文档告知 skill。
- **加载**:P0 一并询问"是否有平台变量文档?",有则读取,得到"变量名 + 替换语义"清单。
- **应用**:主要在 P3 生成 SQL 时,在日期/增量过滤处**主动套用**变量(如 `WHERE tx_date = '${TX_DATE}'`),并向用户说明所用变量及含义、请其确认。也可能影响 P2(全量 vs 增量装载的步骤设计)。
- skill 自带 `references/platform-variables-example.md`,用 `${TX_DATE}` 示范这类文档的形态与约定;用户的真实文档照此结构提供即可。
